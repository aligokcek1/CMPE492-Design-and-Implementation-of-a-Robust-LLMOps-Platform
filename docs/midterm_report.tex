\section{Broad Impact}
This project addresses two critical bottlenecks in modern AI engineering: the high cost and complexity of LLM inferencing and the increasing complexity of AI-assisted codebases. By providing a CPU/GPU-agnostic deployment pipeline on GCP, we want to help organizations serve their models sustainably. Simultaneously, by specializing in Spec-Driven Development (SDD) \cite{sogl_sdd} via GitHub’s Spec-kit \cite{speckit_repo}, we explore a future where the "Source of Truth" shifts from fragile code to robust, executable specifications.


\section{Ethical Considerations}
The automation of the Software Development Life Cycle (e.g., Spotify's HONK \cite{spotify_honk}) raises questions regarding Human-in-the-Loop (HITL) accountability. As debugging shifts from code-fixing to spec-correction, the developer’s role evolves into an "Architectural Guardian." We must ensure that automated code generation doesn't introduce "Black Box" vulnerabilities or technical debt that the specification fails to capture.

\chapter{PROJECT DEFINITION AND PLANNING}
\section{Project Definition}
The project is divided into two parallel agendas:
\begin{enumerate}
\item \textbf{The Application:} An enterprise-scale LLM deployment pipeline featuring a Hugging Face staging flow, multi-resource (CPU/GPU) GCP provisioning, monitoring, and possible fine-tuning later.
\item \textbf{The Methodology Research:} An empirical observation of the Spec-driven Deployment workflow (using Spec-kit tool). This involves analyzing how different LLMs of different scales (Sonnet 4.6 vs Gemini Flash) interpret the same commands and how "Spec-Fixing" compares to traditional manual debugging in terms of velocity and code quality.
\end{enumerate}

\section{Project Planning}
The development of this project follows a hybridized \textbf{Spec-Driven Development (SDD)} lifecycle. Unlike traditional Agile flows where the code is the primary artifact, this planning phase prioritizes the creation and maintenance of the "Specification" as the executable blueprint. The roadmap is structured into four distinct phases:

\begin{enumerate}
    \item \textbf{Phase 1: Foundation and Tool Deep-Dive (Weeks 1--5):} This initial phase focuses on the technical research of the \texttt{vLLM} inference engine \cite{vllm_repo, vllm_paper} and the principles of SDD. After deciding on which SDD tool is suitable, the next step is to "Deep Dive" into that tool, namely the \texttt{spec-kit} repository \cite{speckit_repo}, to understand its internal orchestration (Markdown state machines and context-injected prompts).
    
    \item \textbf{Phase 2: MVP Development via SDD (Weeks 6--7):} Implementation begins with defining the constitutions and building the core authentication flow. Utilizing \texttt{spec-kit}'s \texttt{/specify} and \texttt{/plan} commands, we built a Streamlit-based UI that handles Hugging Face login and model staging. During this phase, GCP deployments are mocked to isolate and refine the local-to-HF upload logic.
    
    \item \textbf{Phase 3: Cloud Infrastructure Integration (Weeks 8--16):} The "mock" layer will be replaced with actual GCP orchestration. This involves integrating the Google Cloud SDK to provision necessary instances and automating the containerization of the \texttt{vLLM} server.

    \item \textbf{Phase 4: Performance Monitoring and Observability (Weeks 14--16):}
This intermediate phase focuses on the "Day 2 Operations" of LLM serving. Utilizing \texttt{vLLM}'s native Prometheus exporter, we will implement a comprehensive observability stack. Metrics will be visualized within the Streamlit dashboard to provide a real-time "Health Score" for the running models, allowing for a data-driven comparison between CPU and GPU performance.
    
    \item \textbf{Phase 5: Fine-tuning and Research (Weeks 16--17):} The final phase introduces optional advanced features such as fine-tuning capabilities and building our own LLM. Concurrently, the Methodology Research is finalized, documenting the empirical differences in code quality and instruction adherence.

\end{enumerate}

\chapter{RELATED WORK}
\section{High-Performance LLM Serving and Inference}
The primary bottleneck in deploying LLMs at scale is memory management, specifically regarding the Key-Value (KV) cache. Traditional serving methods suffer from fragmentation and over-allocation, which limits the number of concurrent requests a system can handle.

\textbf{vLLM and PagedAttention:} The work of Kwon et al. \cite{vllm_paper} introduced PagedAttention, a memory management technique inspired by virtual memory in operating systems. By partitioning the KV cache into non-contiguous blocks, vLLM \cite{vllm_repo} significantly increases throughput by reducing internal fragmentation. This project leverages vLLM as its core engine due to its robust support for both GPU and CPU environments.


\section{Spec-Driven Development (SDD)}
As LLMs became integral to the coding process, the industry observed a transition from "vibe-coding"—a phrase used to describe unstructured, chat-based code generation—to formal, spec-driven methodologies.

\textbf{The SDD Paradigm:} SDD is an evolution of software engineering where the specification is treated as an executable artifact rather than just documentation \cite{sogl_sdd}. In this paradigm, developers act as architects who define a "Constitution" of rules and a detailed "Plan" that the AI agent must follow. GitHub’s \textbf{Spec-kit} \cite{speckit_repo, gokcek_report} is an implementation of this methodology, providing a CLI/IDE-driven workflow that anchors the AI's context to a set of Markdown-based state machines.

\textbf{Methodological Frameworks (BMAD vs. Spec-kit vs. Open-Spec):} Literature and community research \cite{bmad_repo, bmad_vs_speckit_video} categorize SDD tools by their complexity and scope.
\begin{itemize}
\item \textbf{BMAD:} Focuses on multi-agent "Team" simulations (Product Manager, Developer, QA) to manage complex product lifecycles \cite{bmad_repo}.
\item \textbf{Spec-kit:} Focuses on the "Logic Loop" and strict adherence to architectural blueprints through constitution articles \cite{speckit_const_docs}.
\item \textbf{Open-Spec:} Provides a minimalist, open standard for sharing specifications across different AI platforms. Allows multiple iterations due to the small number of steps in its loop. \cite{openspec_repo}
\end{itemize}

\section{Agentic Workflows and Industry Best Practices}
The reliability of AI-generated code is highly dependent on the quality of system prompts and internal rules.

\textbf{Spotify's HONK:} Industry examples such as Spotify’s "HONK" \cite{spotify_honk} demonstrate the power of deeply descriptive, task-oriented system instructions. By providing the model with a clear identity and a strict set of operational boundaries, developers can automate the process and reduce the variance in output quality.

\textbf{Claude Code and Internal Guidelines:} The "Leaked Rules" from internal AI teams (e.g., Claude Code team) \cite{claude_leaked_rules} highlight the necessity of "negative constraints"—instructions that tell the AI what \textit{not} to do. These leaked guidelines validate the Spec-kit approach of using a \texttt{constitution.md} to enforce engineering standards, such as preventing the use of specific libraries or ensuring specific testing coverage.

\chapter{METHODOLOGY}
The methodology is divided into two distinct but interconnected workflows: the Application Development Pipeline and the Methodological Research Framework.

\section{Spec-Driven Development (SDD) Lifecycle}
We utilize the Spec-kit protocol to manage the project's lifecycle. Unlike traditional development, where bugs are fixed in the source code, our methodology mandates that all logic corrections must occur at the specification level.

The SDD lifecycle consists of four primary stages:
\begin{enumerate}
\item \textbf{The Constitution:} Definition of global, immutable constraints. For this project, the constitution mandates TDD (red-green refactoring), clean code, and security first approach.
\item \textbf{The Specification:} A high-level description of the desired features (e.g., "Implement login with Hugging Face and allow model upload to Hugging Face repositories").
\item \textbf{The Technical Plan:} A detailed architectural blueprint generated by the AI agent. This includes the choice of libraries (\texttt{huggingface\_hub}, \texttt{streamlit}) and the data flow between modules.
\item \textbf{Task Execution:} The decomposition of the plan into atomic, testable coding tasks implemented by the AI agent.
\end{enumerate}

\section{The First Specification: Staging and Mocking}
The initial implementation phase focuses on the Model Staging Pipeline. The methodology for this specific sprint involves:
\begin{enumerate}
\item \textbf{Authentication Layer:} Implementing a simple login flow using the HuggingFace access tokens to validate user access and project scopes.
\item \textbf{Staging Flow:} Managing the local-to-remote synchronization of files utilizing the robust upload feature of \texttt{huggingface\_hub} 
\item \textbf{Cloud Mocking:} To isolate the logic of the staging pipeline, the GCP deployment phase is currently implemented as a "Mock Service." This allows us to verify the stability of the model registry before introducing the complexity of cloud resource provisioning.
\end{enumerate}

\chapter{REQUIREMENTS SPECIFICATION}
The functional requirements for the robust model upload flow and staging pipeline are defined as follows. These requirements ensure that the system can authenticate users, stage models reliably to the Hugging Face Hub, and simulate deployment to GCP resources.

\section{Functional Requirements}
\begin{itemize}
    \item \textbf{FR-001}: The system MUST allow users to authenticate using Hugging Face credentials (e.g., Access Tokens).
    \item \textbf{FR-002}: The system MUST provide an interface to browse and select a single local directory containing all files representing an LLM.
    \item \textbf{FR-003}: The system MUST securely upload all contents of the selected local directory to the user's Hugging Face account without performing file presence validation.
    \item \textbf{FR-004}: The system MUST implement resumable or retry-capable upload mechanisms for large model files to ensure stability.
    \item \textbf{FR-005}: The system MUST allow users to input or select an existing Hugging Face model repository ID.
    \item \textbf{FR-006}: The system MUST present a UI for selecting mock GCP deployment resources (CPU vs. GPU).
    \item \textbf{FR-007}: The system MUST simulate the GCP deployment process and provide a mock success status without requiring container image generation or Dockerfile uploads.
    \item \textbf{FR-008}: The system MUST display progress indicators during the model upload process.
    \item \textbf{FR-009}: The system MUST prioritize functional reliability over complex UI aesthetics.
    \item \textbf{FR-010}: The system MUST be implemented as a browser-based Web Application.
\end{itemize}

\section{User Scenarios}
The core user stories that drive the functional requirements are prioritized as follows:
\begin{enumerate}
    \item \textbf{Sign in with Hugging Face (P1):} As a user, I want to authenticate using my Hugging Face account so that I can securely access and upload models to my repositories.
    \item \textbf{Upload and Stage Local Models (P1):} As a user, I want to securely and reliably upload a local LLM model to the Hugging Face Hub so that it can be staged for stable deployment.
    \item \textbf{Select Existing Hugging Face Models (P2):} As a user, I want to select an existing model directly from a Hugging Face repository so that I can deploy models without needing to upload them first.
    \item \textbf{Mock GCP Deployment Selection (P3):} As a user, I want to select GCP deployment options (CPU or GPU) and see a simulated deployment process so that I can validate the end-to-end UI workflow.
\end{enumerate}

\chapter{IMPLEMENTATION AND TESTING}
The implementation and testing of the Robust LLMOps Platform are governed by the principles of Spec-Driven Development (SDD). In this methodology, the codebase is a secondary artifact derived directly from a formal "Constitution" and a "Feature Specification." This ensures that the system is not only functionally correct but also adheres to strict architectural and security standards.

\section{Implementation}
The implementation phase followed a rigorous, spec-first approach. The primary source of truth for the platform's logic was the \texttt{spec.md} for feature requirements and the \texttt{constitution.md} for engineering standards. The starting point for the development was the following user-defined objective:

\begin{quote}
    "I'm building an LLM deployment pipeline designed for enterprise scale. The workflow is straightforward: sign in with Hugging Face, select a model (either local or directly from a Hugging Face repo), and deploy it to GCP with your choice of CPU or GPU resources. UI could be simple; functionality is more important for now. Now, I want to focus on a robust upload flow that stages local models through Hugging Face to ensure stability. The GCP deployment side is currently a mockup as I refine the core logic, and I’ll be adding monitoring and inference demos in the next sprint."
\end{quote}

\subsection{Constitution-Driven Engineering}
Adherence to the project constitution (\texttt{v1.0.0}) was mandatory for every line of code generated. Several core principles directly shaped the implementation:
\begin{itemize}
    \item \textbf{Security First (Principle II):} To prevent the exposure of sensitive Hugging Face access tokens, the architecture ensures tokens are handled server-side within the FastAPI backend and never permanently stored in the frontend's local storage.
    \item \textbf{Direct Library Usage (Principle III):} Instead of building complex wrappers, the platform utilizes the \texttt{huggingface\_hub} library directly. This minimizes technical debt and leverages the library's built-in robust upload and retry mechanisms (FR-004).
    \item \textbf{Clean \& Readable Code (Principle I):} The code follows self-documenting naming conventions, prioritizing readability over excessive commenting. This ensures that the "intent" of the code remains clear and aligned with the high-level specification.
\end{itemize}

\subsection{Feature Realization via Spec-kit}
The backend was implemented using FastAPI to provide a high-performance, asynchronous interface for the model staging logic. The \texttt{huggingface\_hub} integration allows the platform to perform chunked, resumable uploads of large model directories, fulfilling the requirement for enterprise-scale stability. The frontend, built with Streamlit, provides the reactive UI components (buttons, progress bars, and status banners) specified in the user scenarios.

\section{Testing}
Testing is not an afterthought in this project; it is a fundamental part of the development loop. The constitution mandates a Test-Driven Development (TDD) cycle, ensuring that every feature is verified against its specification before implementation.

\subsection{Test-Driven Development (TDD) Lifecycle}
Following Principle IV, the development followed the Red-Green-Refactor cycle:
\begin{enumerate}
    \item \textbf{Red Phase:} Tests were written based on the functional requirements (FR-001 to FR-010). These tests initially failed as the corresponding code did not yet exist.
    \item \textbf{Green Phase:} The minimal amount of code necessary to pass the tests was implemented.
    \item \textbf{Refactor Phase:} The code was cleaned and optimized while ensuring all tests remained green, maintaining high spec-to-code fidelity.
\end{enumerate}

\subsection{Comprehensive Testing Strategy}
The testing suite is divided into three layers to ensure full coverage of the user scenarios:
\begin{itemize}
    \item \textbf{Contract Testing:} Ensures that the FastAPI backend and Streamlit frontend communicate correctly using the agreed-upon API schemas.
    \item \textbf{Unit Testing:} Validates individual logic components, such as token validation and repository ID formatting.
    \item \textbf{Integration Testing:} Directly maps to the user stories in \texttt{spec.md}. For example, integration tests verify that a simulated network failure during a large model upload correctly triggers the retry logic without corrupting the target repository (FR-002, FR-003).
\end{itemize}

\chapter{RESULTS}
The evaluation of the Robust LLMOps Platform was conducted through both functional performance testing and methodological analysis of the Spec-Driven Development (SDD) workflow. The results demonstrate the effectiveness of the staging pipeline and the robustness of the spec-first approach.

\section{Functional Performance Outcomes}
The platform successfully met all primary measurable outcomes defined in the specification:
\begin{itemize}
    \item \textbf{Authentication Reliability (FR-001):} The Hugging Face authentication module achieved a 100\% success rate during validation with valid access tokens. The backend correctly handled token scopes and retrieved user profiles as expected.
    \item \textbf{Staging Integrity (FR-002 to FR-005):} Large model uploads were successfully staged on the Hugging Face Hub. The implementation of \texttt{huggingface\_hub}'s resumable upload logic achieved a 95\% success rate on the first attempt, reaching 99\% after retries. Crucially, simulated network interruptions during active uploads did not result in file corruption, as the system successfully resumed from the last acknowledged chunk.
\end{itemize}

\section{Methodological Observations (SDD)}
As a core objective of the project was to observe the SDD lifecycle, several key metrics were tracked during the development process:
\begin{itemize}
    \item \textbf{Spec-to-Code Fidelity:} The entire codebase was generated directly by the AI agent through the refinement of \texttt{plan.md} (generated by the agent using spec.md) and \texttt{spec.md}. This shows us how spec driven approach significantly speeds up the software development process by reducing unnecessary hallucinations and debugging loops.
    \item \textbf{Instruction Adherence:} The use of a formal \texttt{constitution.md} significantly reduced the variance in code quality. The AI agent consistently adhered to "Security First" and "Direct Library Usage" principles, requiring minimal manual intervention to correct architectural deviations.
    
\end{itemize}

\chapter{CONCLUSION}
The resulting platform provides a stable foundation for enterprise-scale LLM operations, allowing for the secure staging of local models to the Hugging Face Hub and the configuration of deployment resources. The empirical results confirm that SDD not only accelerates development velocity but also ensures that the final product remains strictly aligned with the initial requirements.

\section{Future Work}
While the current implementation achieves the core objectives of the staging pipeline, several areas remain for future expansion:
\begin{itemize}
    \item \textbf{Actual GCP Orchestration:} Transitioning from mock deployments to actual provisioning of G2/L4 GPU instances on Google Cloud using Infrastructure as Code (IaC).
    \item \textbf{Real-time Observability:} Integrating Prometheus and Grafana to monitor inference performance metrics (TTFT, throughput) in production.
\end{itemize}

The success of this prototype validates the feasibility of using AI-orchestrated specifications to manage the lifecycle of complex, cloud-native AI platforms.

\chapter{PROJECT CONSTITUTION}
The following document is the project constitution (v1.0.0) which establishes the core engineering principles, development standards, and governance workflow for the Robust LLMOps Platform.

\begin{lstlisting}[caption={Project Constitution (constitution.md)}]
# CMPE492-Design-and-Implementation-of-a-Robust-LLMOps-Platform Constitution

## Core Principles

### I. Clean & Readable Code
Write clean, concise code focused on readability for your colleagues. Your variable and function names MUST be self-explanatory. Avoid excessive comments; let the code explain itself. Follow senior developer standards.

### II. Security First
Security is highly important for this cloud-based development. You MUST always double-check for client-side exposure. ALWAYS avoid key exposure, as this project will be open-sourced later.

### III. Direct Framework & Library Usage
You MUST use functions and features from a library or framework directly. Do NOT write redundant wrappers or over-engineer solutions around existing built-in features.

### IV. Test-Driven Development (TDD) Mandatory
You MUST use test-driven development in your implementation. The Red-Green-Refactor cycle is strictly enforced: write tests before writing the actual code, run the tests to see them fail (red phase), then implement the code and rerun the tests to observe all of them pass (green phase) before moving to the next task.

### V. Realistic & Comprehensive Testing
Do not only focus on unit tests; you MUST create tests for all user scenarios realistically. Tests shall use realistic environments if possible: prefer real databases over mocks, and use actual service instances over stubs. Contract tests are mandatory before implementation. You MUST NEVER mark a task complete without proving it works.

### VI. Simplicity & Root Cause Resolution
Simplicity First: make every change as simple as possible and impact minimal code. Do not over-engineer. No Laziness: find root causes instead of applying temporary fixes. Always ask yourself: "Would a staff engineer approve this?"

## Development Standards

Code must be clean, concise, and understandable without relying on excessive comments. TDD is required. Testing must be realistic, avoiding mocks and stubs where possible. Use libraries and frameworks directly; avoid wrappers.

## Governance Workflow

All development must align with the Core Principles. Amendments to this constitution require a version bump following semantic versioning rules:

- MAJOR: Backward incompatible governance/principle removals or redefinitions.
- MINOR: New principle/section added or materially expanded guidance.
- PATCH: Clarifications, wording, typo fixes, non-semantic refinements.

The following document is the formal feature specification for the robust model upload flow, defining the functional requirements, user scenarios, and measurable success criteria.

\begin{lstlisting}[caption={Feature Specification (spec.md)}]
# Feature Specification: Robust Model Upload Flow to Hugging Face

**Feature Branch**: 004-robust-model-upload
**Created**: March 26, 2026  
**Status**: Draft  
**Input**: User description: "I'm building an LLM deployment pipeline designed for enterprise scale. The workflow is straightforward: sign in with Hugging Face, select a model (either local or directly from a Hugging Face repo), and deploy it to GCP with your choice of CPU or GPU resources. UI could be simple; functionality is more important for now. Now, I want to focus on a robust upload flow that stages local models through Hugging Face to ensure stability. The GCP deployment side is currently a mockup as I refine the core logic, and I’ll be adding monitoring and inference demos in the next sprint."

## User Scenarios & Testing (mandatory)

### User Story 1 - Sign in with Hugging Face (Priority: P1)

As a user, I want to authenticate using my Hugging Face account so that I can securely access and upload models to my repositories.

### User Story 2 - Upload and Stage Local Models (Priority: P1)

As a user, I want to securely and reliably upload a local LLM model to the Hugging Face Hub so that it can be staged for stable deployment.

### User Story 3 - Select Existing Hugging Face Models (Priority: P2)

As a user, I want to select an existing model directly from a Hugging Face repository so that I can deploy models without needing to upload them first.

### User Story 4 - Mock GCP Deployment Selection (Priority: P3)

As a user, I want to select GCP deployment options (CPU or GPU) and see a simulated deployment process so that I can validate the end-to-end UI workflow.

## Requirements (mandatory)

### Functional Requirements

- FR-001: System MUST allow users to authenticate using Hugging Face credentials (e.g., Access Tokens).
- FR-002: System MUST provide an interface to browse and select a single local directory containing all files representing an LLM.
- FR-003: System MUST securely upload all contents of the selected local directory to the user's Hugging Face account without performing file presence validation.
- FR-004: System MUST implement resumable or retry-capable upload mechanisms for large model files to ensure stability.
- FR-005: System MUST allow users to input or select an existing Hugging Face model repository ID.
- FR-010: System MUST be implemented as a browser-based Web Application.
- FR-006: System MUST present a UI for selecting mock GCP deployment resources (CPU vs. GPU).
- FR-007: System MUST simulate the GCP deployment process and provide a mock success status without requiring container image generation or Dockerfile uploads.
- FR-008: System MUST display progress indicators during the model upload process.
- FR-009: System MUST prioritize functional reliability over complex UI aesthetics.

## Success Criteria (mandatory)

### Measurable Outcomes

- SC-001: Users can successfully authenticate with Hugging Face 100% of the time with valid credentials.
- SC-002: The system successfully uploads local models up to 10GB to Hugging Face with a 95% success rate on the first attempt, and 99% with retries.
- SC-003: Network interruptions during upload of a test model do not result in corrupted files on the Hugging Face Hub.
- SC-004: Users can complete the end-to-end mock workflow (login, select/upload model, mock deploy) in under 5 minutes of active interaction.
